# 0114 Java Security Methods

## 회원가입과 인증
### 1. 회원가입 과정
1. 사용자가 회원가입 정보를 입력.
2. 서버는 해당 정보를 데이터베이스(DB)에 저장.
3. 이후 인증 과정에서 DB에 저장된 정보를 사용하여 비교.

---

### 2. 쿠키를 사용한 인증 방식
- **작동 원리**
  - 서버에서 클라이언트로 쿠키에 특정 문자를 저장.
  - 클라이언트가 GET이나 POST 요청을 보낼 때, 해당 쿠키를 포함하여 전송.
- **장점**
  - 간단한 구현.
- **단점**
  - 쿠키가 탈취되면 보안 문제가 발생할 수 있음.

---

### 3. 세션(Session) 방식
- **작동 원리**
  - 사용자가 로그인하면 서버에서 `Session ID`를 생성.
  - `Session ID`, 사용자 이름, 유효기간을 서버에 저장.
  - 클라이언트는 `Session ID`를 쿠키나 요청 헤더에 포함하여 서버에 전송.
- **유효성 검증**
  - 사용자의 모든 GET, POST 요청마다 서버는 `Session ID`를 확인.
  - DB에 저장된 세션 정보를 기반으로 사용자 인증을 수행.
- **장점**
  - 비교적 안전하게 사용자 상태를 유지.
- **단점**
  - 사용자의 요청마다 DB를 자주 확인하므로 성능 저하 가능.

---

### 4. 토큰(Token) 방식 (예: JWT)
- **작동 원리**
  - 사용자가 로그인에 성공하면 서버는 사용자 정보(ID, 로그인 날짜, 유효기간 등)를 암호화하여 토큰(JWT)을 발급.
  - 클라이언트는 해당 토큰을 요청 헤더에 포함하여 서버로 전송.
- **유효성 검증**
  - 서버는 토큰의 유효성(JWT 서명, 만료 시간 등)을 확인.
  - DB를 직접 확인하지 않고 인증 가능.
- **장점**
  - DB를 자주 확인하지 않아 성능 향상.
  - 분산 환경(예: 마이크로서비스)에 적합.
- **단점**
  - 토큰이 중간에서 가로채기(Man-In-The-Middle) 공격을 받으면 막기 어려움.
  - 토큰이 만료되기 전까지 탈취된 토큰을 악용할 가능성.

---

### 5. OAuth
- **설명**
  - 소셜 로그인 등 제3자 인증 시스템을 사용.
  - 사용자가 특정 서비스(예: Google, Facebook) 계정으로 로그인하면, 해당 서비스에서 인증 후 사용자 정보를 제공.
- **작동 원리**
  1. 클라이언트가 인증 요청.
  2. 인증 서버가 인증 후 액세스 토큰 발급.
  3. 클라이언트는 액세스 토큰을 사용하여 사용자 정보 요청.
- **장점**
  - 사용자 편의성 증가 (기존 계정 활용).
  - 인증 로직을 외부에 위임하여 개발 부담 감소.
- **단점**
  - 제3자 서비스 의존성 증가.

---

## 추가 보안 고려 사항
### 1. HTTPS
- 통신 암호화를 위해 HTTPS를 사용하여 중간자 공격(Man-In-The-Middle)을 방지.

### 2. CSRF 보호
- CSRF 토큰을 활용하여 사용자가 의도하지 않은 요청 방지.

### 3. XSS 방지
- 사용자 입력값을 철저히 검증 및 HTML 태그 필터링.

### 4. 데이터 암호화
- 비밀번호는 반드시 해시 알고리즘(예: BCrypt, SHA-256)으로 암호화하여 저장.

### 5. 토큰 만료 정책
- 토큰 유효기간을 짧게 설정하고, 정기적으로 갱신.

| 인증 방식  | 장점                                | 단점                                      |
|-------------|-------------------------------------|-------------------------------------------|
| 쿠키        | 간단한 구현                        | 탈취 시 보안 문제 발생 가능               |
| 세션        | 비교적 안전, 상태 유지 가능        | DB 접근이 많아 성능 저하 가능             |
| 토큰 (JWT)  | 성능 향상, 분산 시스템에 적합      | 토큰 탈취 시 위험                         |
| OAuth       | 사용자 편의성, 개발 부담 감소      | 제3자 서비스 의존성                       |
