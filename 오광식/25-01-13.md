# 실시간 동시 편집

Google Docs나 Figma, VSCode 의 LiveShare 등은 온라인에서 하나의 문서를 동시에 편집할 수 있고, 실시간으로 여러 유저의 편집 내용이 각자 편집 화면에 바로 반영됩니다. 이렇듯 실시간 동시 편집 기술을 제공하는 여러 서비스들이 많아지고 있습니다.

하지만 동시 편집은 인간의 눈엔 직관적인 것과 다르게, 구현상에서는 분산된 환경에서의 동시성은 복잡한 문제를 가지고 있으며, 그런 모호함을 해결하기 위한 여러 기술들이 대두되고 있습니다.

## OT(Operational Transformation)

OT는 입력한 순서에 따라 서버가 이를 적절히 변형하여 전달하는 방식입니다. OT 는 시간상의 순서를 고려해 우선 순위를 부여하고, 앞에서 적용할 변경사항이 다음 순위의 변경사항을 보정하는 정보로 사용됩니다.

유저1이 맨 앞에 'C'를 입력한 변경사항과 유저2의 첫 문자 삭제 변겨사항을 서버가 각각 수집합니다. 이후 변경사항이 생긴 시간에 맞춰 정렬합니다. 'C' 문자 삽입이 시간적으로 앞서 발생한 변경사항이라면, 'C' 문자 입력으로 인해 이후 문자들의 인덱스가 1씩 뒤로 밀리게 됩니다. 결과적으로 첫 번째 문자 삭제 변경사항은 두 번째 문자 삭제로 인덱스가 보정되어 전달됩니다.

유저1과 유저2이 똑같은 위치의 문자를 삭제한 경우에 보정 결과는 변경사항 없음으로 전달됩니다.

OT방식은 꽤 직관적이여서 초기의 동시 편집 기능 도입때 많이 사용되었습니다. 아직도 Google Docs나 Microsoft Office가 OT 방식을 사용하고 있습니다.

하지만 OT 방식의 가장 큰 문제점은 중앙 집중 처리 방식입니다. 즉 중앙에서 변경사항을 보정해줄 서버가 필요합니다. 이는 트래픽이 몰렸을 때 서버에 과부하가 올 수 있다는 단점이 존재합니다. 이는 비용 측면에서도 좋지 않으며, 사용자 입장에서도 서버 연산 후 응답을 받는 시간이 필요하므로 사용성이 떨어집니다.

## CRDT(Conflict free Replicated Data Types)

동시 편집 기술 초기에는 OT가 주류였으나, 현재에는 CRDT 방식이 인기를 얻고 있습니다. CRDT의 장점은 중앙 서버가 필요 없고, 문서를 편집하는 유저들끼리만 데이터를 교환하면 되므로 속도가 빠르고 서버의 부하를 줄이는 효과가 있었습니다.

OT는 편집 시간과 인덱스를 기반으로 변경사항을 보정하는 것과 다르가, CRDT는 스트림 상의 각 문자에 고유한 id를 부여하고, 이를 기반으로 보정을 정의한다는 차이점이 있습니다.

CRDT는 변경사항을 보정할 필요 없이 바로 다른 유저의 문서에 적용됩니다. 시간 순서에 대한 우선순위 없이 정확한 위치에 해당 문자를 삽입할 수 있습니다. 유저1이 'C'를 추가하는 경우, 새로운 아이디를 부여하고 문자를 id가 1인 문자의 위치에 추가합니다. 그리고 유저2의 환경에서 변경사항을 받아 적용할 수 있습니다. 유저2가 id가 1인 문자를 삭제하는 변경도 마찬가지로 유저1에서 변경사항을 적용할 수 있게됩니다.

유저1과 유저2가 동일한 위치의 문자를 삭제하더라도 인덱스가 아닌 id를 기반으로 하기에 이미 삭제된 id는 존재하지 않아서 중복 삭제도 문제가 되지 않습니다.

이런 CRDT에서도 OT와 비교한 단점이 존재합니다.

OT보다 많은 메모리를 사용합니다. 고유 id 값을 저장하는 메모리와 이를 트리구조로 관리하기 위한 메모리를 필요로 하기 때문에, 일반적인 문서보다 2~3배 더 큰 메모리를 사용합니다.

Peer-to-Peer 통신이 항상 가능하지는 않습니다.

CRDT가 시간을 기반으로 하는것이 아닌 id를 기반으로 하기 때문에, 실시간성이 모호해지고 동기화 결과 문자열이 섞여 의도하지 않은 결과물이 나올 수 있습니다.

하지만 이런 단점들도 현재에는 여러 기술들이 나오면서 개선되고 있습니다.

- 속도 : 요즘의 CRDT (Automerge / RGA or y.js / YATA) 들은 log(n) 수준의 조회 가능합니다.
- 용량 : Martin 의 Columnar 인코딩은 문서 크기의 1.5~2배 크기 정도로 저장이 가능합니다.
- 기능 : 이론적으로는 Rewinding 및 replaying이 가능합니다.
  복잡도 : 구현체 크기가 OT보다 CRDT가 크긴 하겠지만 큰 차이는 없습니다.
  또한 CRDT만의 장점들 또한 명확하게 존재합니다.

중앙 서버가 필요 없고 유저들 끼리 변경사항을 주고 받을 수 있어서 속도가 빠릅니다.
오프라인 환경에서도 문서를 편집하고, 이후 온라인이 되었을때 변경사항 업로드가 가능합니다. 즉 네트워크에 구애를 받지 않습니다. (git 처럼 사용 가능)

## Yorkie

**Yorkie**는 실시간 협업 애플리케이션을 구축할 수 있도록 설계된 오픈소스 CRDT(Conflict-free Replicated Data Types) 기반 프레임워크입니다. Yorkie는 실시간으로 데이터 동기화가 필요한 여러 사용자가 동시에 작업하는 환경(예: 화이트보드, 문서 편집기, 드로잉 앱 등)을 간단히 구현할 수 있도록 도와줍니다.

### Yorkie의 주요 특징

1. CRDT 기반 데이터 모델
   CRDT는 데이터 충돌 없이 여러 클라이언트에서 독립적으로 데이터를 편집하고 동기화할 수 있도록 하는 데이터 구조입니다.
   Yorkie는 CRDT를 기반으로 실시간 협업에서 발생할 수 있는 데이터 충돌을 자동으로 처리합니다.
2. 클라이언트-서버 아키텍처
   Yorkie는 서버-클라이언트 아키텍처를 채택하여, 클라이언트는 로컬에서 데이터를 편집하고 변경 내용을 Yorkie 서버와 동기화합니다.
   클라이언트는 Yorkie 서버에 연결하여 데이터를 공유하며, 서버는 변경 사항을 다른 클라이언트에 브로드캐스트합니다.
3. 실시간 동기화
   Yorkie는 클라이언트가 로컬에서 작업한 데이터를 서버와 동기화하고, 다른 클라이언트에 실시간으로 변경 사항을 전달합니다.
   오프라인 모드에서도 로컬 데이터를 편집할 수 있으며, 다시 연결되면 자동으로 동기화됩니다.
4. 데이터 구조
   Yorkie는 **문서(Document)**라는 개념을 사용하여 데이터를 관리합니다.
   문서는 JSON과 유사한 데이터 구조로, 다양한 데이터를 저장하고 관리할 수 있습니다.
5. 사용 사례
   실시간 협업 편집기: Google Docs와 같은 문서 편집기.
   화이트보드 애플리케이션: 실시간 드로잉 및 다이어그램 작성.
   데이터 시각화 도구: 여러 사용자가 데이터를 동시에 편집하고 시각화.
   멀티플레이어 게임: 상태 동기화와 충돌 관리가 필요한 게임.

### Yorkie의 구성 요소

1. Yorkie 서버
   Yorkie의 핵심 컴포넌트로, 클라이언트 간 데이터를 동기화하고 관리합니다.
   모든 협업 작업이 서버를 통해 전달됩니다.
   클라우드 환경 또는 로컬에서 실행할 수 있습니다.
2. Yorkie 클라이언트
   각 사용자(또는 디바이스)에 설치되는 클라이언트 라이브러리입니다.
   사용자는 로컬에서 데이터를 편집하고 서버와 동기화합니다.
   주요 기능:
   문서(Document) 관리
   실시간 데이터 동기화
3. 데이터 저장소
   Yorkie는 서버의 상태를 관리하기 위해 MongoDB를 기본 데이터 저장소로 사용합니다.
   영구적인 데이터 저장 및 복구 기능을 제공합니다.

### Yorkie의 동작 흐름

1. 문서 생성 및 연결:
   클라이언트는 Yorkie 서버에 연결하고, 협업할 문서를 생성하거나 기존 문서를 가져옵니다.

2. 로컬 데이터 편집:
   클라이언트는 로컬에서 데이터를 편집하며, 이 작업은 CRDT를 통해 충돌 없이 관리됩니다.

3. 동기화:
   클라이언트는 로컬 변경 사항을 Yorkie 서버로 전송합니다.
   서버는 변경 사항을 다른 클라이언트로 브로드캐스트하여 실시간 동기화를 수행합니다.

4. 저장:
   모든 문서 상태는 MongoDB와 같은 영구 저장소에 저장됩니다.

### Yorkie의 주요 장점

1. 실시간 협업에 특화:
   충돌 없는 실시간 데이터 동기화 기능 제공.
   여러 사용자가 동시에 작업해도 데이터 손실이나 충돌이 발생하지 않음.

2. 확장성:
   클라이언트-서버 아키텍처와 MongoDB 기반으로 대규모 프로젝트에서도 안정적으로 작동.

3. 간편한 설정:
   Yorkie 서버는 Docker 컨테이너로 손쉽게 배포 가능.
   클라이언트 라이브러리 사용이 간단하며, 다양한 언어와 프레임워크에서 사용 가능(JavaScript, TypeScript 등).

4. 오프라인 지원:
   클라이언트는 오프라인 상태에서도 작업 가능하며, 다시 연결되면 동기화가 자동으로 이루어짐.

5. 오픈소스:
   Yorkie는 완전한 오픈소스 프로젝트로, 커스터마이징 및 확장이 용이.

## tldraw

tldraw는 오픈소스 드로잉 및 다이어그램 라이브러리로, 간단한 그래픽 에디터부터 실시간 협업 도구까지 다양한 애플리케이션을 개발할 수 있는 강력한 도구입니다. React와 TypeScript로 작성되었으며, React 컴포넌트로 손쉽게 통합할 수 있어 빠르게 UI를 구축할 수 있습니다.

### tldraw의 주요 특징

1. 직관적이고 강력한 드로잉 도구
   도형, 텍스트, 이미지 삽입: 선, 사각형, 원, 텍스트 및 이미지 등 다양한 요소를 편집 가능.
   레이어 및 그룹화 지원: 개별 요소를 레이어로 관리하거나 그룹으로 묶을 수 있음.
   스냅 및 정렬: 도형을 정렬하거나 이동할 때 스냅 기능을 제공해 정교한 작업 가능.

2. 실시간 협업 지원
   tldraw는 실시간 협업에 필요한 상태 관리와 데이터 동기화를 위한 CRDT 기반 구조를 지원합니다.
   내장된 tldraw/sync 라이브러리를 사용하면 간단히 실시간 협업을 구현할 수 있습니다.

3. React 통합
   React 컴포넌트로 제공되므로, 기존 React 프로젝트에 쉽게 추가 가능.
   확장성과 유연성을 고려한 구조로, 도구나 UI를 자유롭게 커스터마이징할 수 있음.

4. 오픈소스 및 확장성
   MIT 라이선스로 제공되어 상업적 프로젝트에도 사용 가능.
   고도로 커스터마이징 가능한 API를 제공하여 맞춤형 드로잉 도구 개발 가능.

### 장점

1. 간편한 React 통합
   tldraw는 React 컴포넌트 형태로 제공되어 기존 프로젝트에 바로 통합 가능.
   상태 관리와 이벤트 처리가 React 친화적으로 설계됨.

2. 실시간 협업 지원
   실시간 협업을 손쉽게 구현할 수 있도록 CRDT 기반의 상태 동기화 기능 내장.
   tldraw/sync 라이브러리를 사용하면 전용 서버 없이도 협업 기능 구현 가능.

3. 고도로 커스터마이징 가능
   도구, UI, 이벤트를 자유롭게 확장하고 수정할 수 있어 맞춤형 요구사항에 적합.
   예를 들어, 도형 추가 버튼이나 사용자 지정 툴을 쉽게 구현 가능.

4. 오픈소스
   무료로 사용 가능하며, 소스 코드를 분석하거나 수정하여 프로젝트에 맞게 최적화 가능.

## Yorkie + tldraw

### Yorkie의 역할

CRDT(Conflict-free Replicated Data Types)를 활용하여 여러 사용자의 데이터를 실시간으로 동기화.
충돌 없는 데이터 병합을 보장하며, 오프라인 상태에서도 작업 가능.

### tldraw의 역할

드로잉 도구와 사용자 인터페이스(UI)를 제공.
Yorkie와 결합해 실시간 드로잉 상태를 동기화.

### Yorkie + tldraw 통합

A. Yorkie 클라이언트 설정

기본 Yorkie 클라이언트 생성
Yorkie 클라이언트와 문서를 설정하고 tldraw 상태를 동기화합니다.

B. 주요 동작 설명

1. Yorkie 문서 초기화:
   tldraw-room이라는 협업 방 이름으로 Yorkie 문서를 생성합니다.
   문서의 루트 데이터(root.tldraw)를 tldraw의 상태와 연결합니다.

2. tldraw 상태를 Yorkie로 동기화:
   app.store.listen을 사용해 tldraw의 상태 변경 사항을 감지하고, 이를 Yorkie 문서에 업데이트합니다.

3. Yorkie 상태를 tldraw로 반영:
   Yorkie 문서가 업데이트될 때마다 app.store.deserialize를 호출하여 tldraw의 상태를 갱신합니다.

4. 실시간 동기화:
   Yorkie 클라이언트는 서버와 연결되어 모든 클라이언트 간 상태 변경 사항을 동기화합니다.
